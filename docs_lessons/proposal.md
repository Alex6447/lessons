## 1. Краткое резюме (что мы строим)

**Одной фразой:** маркетплейс + рабочее пространство для репетиторов, где ИИ помогает
собирать индивидуальные курсы, а ученики проходят их в личном кабинете с оплатой,
видеосвязью и общением.

В продукте уживаются **три разных продукта**, и это главный риск по объёму. Их нужно
честно развести по приоритету:

| # | Продукт внутри идеи | Кому | Приоритет для MVP |
|---|---------------------|------|-------------------|
| A | **Кабинет репетитора** (ученики, материалы, уроки, оплата) | Репетитор | 🟢 Ядро MVP |
| B | **Витрина / маркетплейс** (демо-материалы «как YouTube», заявки) | Новый ученик | 🟡 Лёгкая версия в MVP |
| C | **ИИ-генератор курсов и материалов** (диаграммы, схемы, программы) | Репетитор | 🟡 Узкий сценарий в MVP |
| D | **Видеочаты + мессенджер + рассылки** | Оба | 🔴 После MVP (интеграция, не своя разработка) |

> **Главная рекомендация:** не строить всё сразу. MVP = A целиком + тонкие срезы B и C.
> D на старте закрываем интеграцией (Jitsi/100ms/Daily для видео, готовый чат), а не
> собственной разработкой — иначе MVP не выйдет на рынок за разумный срок.

## 2. Предложения по продукту (что добавить / уточнить к idea.md)

### 2.1 Определиться с бизнес-моделью заранее
От неё зависит вся архитектура оплат. Варианты:
- **SaaS-подписка с репетитора** (он платит за рабочее место, ученики платят ему напрямую) —
  проще юридически, мы не держим чужие деньги.
- **Комиссия с транзакций** (маркетплейс берёт %) — сложнее: эскроу, возвраты, налоги,
  мы становимся платёжным посредником.
- **Гибрид** — бесплатный вход + комиссия + платные про-функции.

  Для MVP рекомендую **SaaS-подписку + опционально комиссию позже**. Это снимает большую
  часть юридической нагрузки на старте.

### 2.2 Чётко описать роли и их сценарии
- **Репетитор**: регистрация → витрина-профиль → создание курса (вручную или с ИИ) →
  приглашение/приём учеников → проведение уроков → выставление оплаты → рассылки.
- **Ученик**: поиск на витрине → демо-материалы → заявка/оплата → кабинет → прохождение
  курса → уроки/созвоны с репетитором.
- **Админ платформы**: модерация витрины, поддержка, биллинг, аналитика.
- (позже) **Родитель** — отдельная роль для школьников: оплата и прогресс ребёнка.

### 2.3 «Официальные программы страны» — осторожно
Пункт про «любые официально существующие программы» — это огромный и юридически
чувствительный объём контента (ФГОС и т.п.). Предложение: в MVP **не загружать готовые
гос-программы**, а дать ИИ-помощнику генерировать курс *по теме/цели*, опираясь на
структуру (классы, уровни). Полные официальные программы — отдельный этап с проверкой
методистами.

### 2.4 ИИ-помощник — сузить до одного сильного сценария
«Диаграммы, рисунки, схемы» — это разные технологии. Для MVP предлагаю один killer-сценарий:

> **«Опиши цель ученика → ИИ собирает структуру курса (модули → уроки → задания) +
> текстовые материалы + схемы в виде диаграмм (Mermaid)».**

Mermaid даёт диаграммы/схемы из текста почти бесплатно. Картинки/иллюстрации — позже.

### 2.5 Что важно не забыть (часто всплывает поздно)
- **Прогресс и аналитика ученика** (без этого платформа = папка с файлами).
- **Расписание и календарь** уроков, напоминания.
- **Хранение и права доступа к материалам** (ученик видит только своё).
- **Мобильность** — ученики придут с телефона. Минимум адаптивная вёрстка.
- **Юридическое**: оферта, политика конфиденциальности, согласие на обработку ПДн,
  работа с несовершеннолетними.

## 3. Предлагаемый технологический стек

Выбран под скорость вывода MVP, найм и зрелость экосистемы.

| Слой | Выбор | Почему |
|------|-------|--------|
| Фронтенд | **Next.js (React) + TypeScript** | SSR для SEO витрины, один язык на фронт/бэк |
| UI | **Tailwind CSS + shadcn/ui** | быстрый современный дизайн, легко делать уникальную тему |
| Бэкенд | **Next.js API routes / NestJS** (по росту) | начинаем монолитом, дробим позже |
| БД | **PostgreSQL** (через Prisma ORM) | реляционка под роли/курсы/платежи |
| Файлы/медиа | **S3-совместимое хранилище** (напр. Cloudflare R2) | материалы, видео демо |
| Аутентификация | **Auth.js / Clerk** | роли, соц-логин, быстро |
| Платежи | **ЮKassa**: подписка + **сплитование** (выплаты репетиторам) | рекуррент за тариф + приём оплат учеников с авторазделением платформа/репетитор, чеки 54-ФЗ, выплаты самозанятым/ИП |
| ИИ | **Anthropic Claude API** | генерация структуры курса и материалов |
| Видео | **внешний провайдер** (Daily / 100ms / Jitsi) | не изобретать WebRTC |
| Реалтайм чат | **Postgres + WebSocket / Supabase Realtime** | мессенджер, статусы |
| Деплой | **Хостинг с РФ-локацией** (VPS/облако в РФ) + управляемый Postgres | хранение ПДн граждан РФ на серверах в РФ (152-ФЗ); Vercel — только если данные ПДн локализованы отдельно |

> Стек зафиксирован под рынок РФ/СНГ (см. раздел 0): ЮKassa, локализация ПДн.

### 3.1 Мультитенантность (с нуля)
Решено стартовать сразу мультитенантом. Подход для MVP — **shared DB, tenant-by-row**:
у всех ключевых таблиц есть `tenantId` (= репетитор/организация), и **каждый** запрос
обязательно фильтруется по нему. Это закрывается на уровне ORM (middleware Prisma или
Row-Level Security в Postgres), чтобы изоляцию нельзя было забыть в отдельном запросе.
Полное разделение БД на тенант — преждевременно, переедем при росте.

## 4. Архитектура и модель данных (черновик)

Ключевые сущности (Prisma-стиль, упрощённо):

```
PlatformSettings (id, key, value, group) — все настраиваемые параметры (см. 4.2), редактируются в админке
Plan (id, name, price, limits{students,courses,storage}, commissionRateOverride?) — тарифы абонплаты
Tenant (id, name, planId, status, commissionRateOverride?) — «рабочее пространство» репетитора
Subscription (id, tenantId, plan, status, provider=ЮKassa, currentPeriodEnd) — абонплата
User (id, tenantId?, role[tutor|student|admin], email, profile)
TutorProfile (userId, tenantId, bio, subjects[], rating, demoMaterials[])
PayoutAccount (id, tenantId, type[самозанятый|ИП|физлицо], inn, юkassa_recipient) — куда платим репетитору
Course (id, tenantId, tutorId, title, goal, level, ageGroup, price, status[draft|published])
Module (id, courseId, order, title)
Lesson (id, moduleId, order, title, content, diagrams, price?)
Assignment (id, lessonId, type, payload)
Enrollment (id, tenantId, studentId, courseId, status, progress%)
Progress (id, enrollmentId, lessonId, status, score)
Payment (id, tenantId, studentId, target[course|lesson], amount, commissionRate, commission, tutorShare, status, provider, receiptUrl) — оплата ученика (ставку фиксируем на момент платежа)
Payout (id, tenantId, tutorId, amount, period, status) — выплата репетитору
LiveSession (id, tenantId, tutorId, students[], scheduledAt, videoRoomUrl)
Message / Thread (мессенджер, рассылки) — в рамках tenant
Lead (id, tenantId, studentContact, status) — заявки от новых клиентов
```

> Два денежных потока: **`Subscription`** = абонплата репетитора нам (рекуррент);
> **`Payment`** = оплата ученика за урок/курс, из которой `commission` идёт платформе,
> а `tutorShare` копится репетитору и выплачивается через `Payout`. `tenantId` есть почти
> везде и **всегда** в фильтре запроса (см. 3.1).

Связи: `Tenant → Users/Courses/...`; `Course → Module → Lesson → Assignment`;
`Student —Enrollment→ Course`; `Progress` висит на enrollment+lesson. Витрина строится
из `published` курсов и `demoMaterials` репетиторов (кросс-tenant, только публичное).

### 4.1 Платёжная модель (гибрид) — что учесть

Принятая модель делает нас **платёжным посредником**, и это самая юридически
чувствительная часть. Ключевые моменты:

- **Настраиваемая ставка комиссии.** Хранится в настройках, не в коде: глобальный дефолт
  `defaultCommissionRate = 3%` (`PlatformSettings`), с возможностью переопределить на
  конкретного репетитора/тариф (`Tenant.commissionRateOverride`). На каждом платеже
  ставка **фиксируется в момент оплаты** (`Payment.commissionRate`), чтобы смена дефолта
  не пересчитывала старые платежи. Менять дефолт можно из админки без релиза.
- **Сплитование платежей.** Оплата ученика сразу делится: комиссия платформы +
  доля репетитора. ЮKassa поддерживает разделение (Сплит/«Безопасная сделка»). Это
  избавляет нас от ручного держания и развода чужих денег.
- **Статус репетитора для выплат.** Чтобы платить репетитору легально, он должен быть
  **самозанятым / ИП / физлицом** с реквизитами (`PayoutAccount`). От этого зависит чек и
  налоги. Большинство репетиторов — самозанятые → удобно через ЮKassa-выплаты самозанятым.
- **Чеки 54-ФЗ с двух сторон.** Чек ученику за оплату и чек/НПД-квитанция за услугу
  репетитора. Нужно заранее понять, кто фискализирует (мы как агент или репетитор сам).
- **Возвраты и споры.** Ученик не пришёл / отменил урок → политика возвратов, частичный
  рефанд, удержание комиссии. Это отдельная логика состояний у `Payment`.
- **Когда списываем комиссию.** За урок (поштучно) или за курс (разово) — это **настройка**
  в админке (4.2), не решение в коде. Дефолт рекомендую «оплата курса целиком» (проще сплит).
- **Юр-обвязка.** Договор-оферта платформы, агентский договор с репетитором, согласие
  на обработку ПДн. Это блокер запуска платежей — закладываем время.

> ⚠️ Этот блок ощутимо увеличивает объём MVP по сравнению с «чистой подпиской». Поэтому в
> плане (раздел 6) приём оплат учеников вынесен в **Этап 2** и допускает запуск ядра (Этап 1)
> **до** готовности эквайринга — репетитор может вести учеников, пока деньги настраиваются.

### 4.2 Админ-панель → вкладка **Settings** (всё настраивается здесь)

**Принцип:** никакие бизнес-параметры не «зашиты» в код. Размер комиссии, тарифы, лимиты,
способ начисления и т.п. — всё хранится в БД (`PlatformSettings` + связанные таблицы) и
редактируется суперадмином во вкладке **Settings** без релиза. Код читает значения из
настроек, дефолты заданы в сидах (напр. комиссия 3%).

Группы настроек на вкладке Settings:

| Группа | Настройки (примеры) | Где хранится |
|--------|---------------------|--------------|
| **Монетизация** | ставка комиссии (дефолт 3%), база начисления (курс / урок), валюта, округление | `PlatformSettings` |
| **Тарифы абонплаты** | список планов: название, цена, лимиты (учеников/курсов/хранилища), переопределение комиссии для плана | `Plan[]` |
| **Платежи** | ключи ЮKassa, режим (тест/боевой), правила сплита, настройки выплат/`Payout` (порог, периодичность) | `PlatformSettings` (секреты — в env/секрет-хранилище) |
| **Возвраты** | окно отмены, правила частичного рефанда, удержание комиссии при возврате | `PlatformSettings` |
| **Фискализация** | кто бьёт чек (агент/репетитор), ставка НДС, тексты чеков 54-ФЗ | `PlatformSettings` |
| **Витрина/контент** | категории/предметы, модерация (вкл/выкл), что показывать на лендинге | `PlatformSettings` / справочники |
| **ИИ** | модель, лимиты генераций на тариф, вкл/выкл функции | `PlatformSettings` |
| **Право/тексты** | ссылки на оферту, политику ПДн, агентский договор | `PlatformSettings` |
| **Брендинг** | логотип, цвета темы, название платформы | `PlatformSettings` |

Реализация:
- Единый сервис `getSetting(key)` с кэшем; запись через Settings UI пишет в БД и сбрасывает кэш.
- **Аудит изменений** настроек (кто/когда/что менял) — важно для денежных параметров.
- Значения, влияющие на деньги, **фиксируются в транзакции** (как `Payment.commissionRate`),
  чтобы изменение настройки не переписывало историю.
- Доступ к Settings — только роль **суперадмин платформы** (не репетитор-tenant).

> Это снимает большинство «открытых вопросов» по монетизации: их не надо решать сейчас в
> коде — закладываем значения по умолчанию и даём менять их в админке.

## 5. Подход к дизайну (как в idea.md — «сначала структура, потом дизайн окон»)

Согласен с порядком из п.7 идеи. Конкретизирую процесс:

1. **Информационная архитектура** — карта экранов и навигация (см. список ниже).
2. **Low-fi вайрфреймы** каждого экрана (Figma или прямо в коде на shadcn-заглушках).
3. **Дизайн-система**: палитра, типографика, компоненты, тёмная/светлая тема.
   Уникальность достигается фирменным акцентом + иллюстрациями, а не переусложнением.
4. **Hi-fi макеты** ключевых экранов → затем реализация.

**Карта основных экранов:**
- Публичные: лендинг, витрина/каталог, страница репетитора, страница курса (демо), регистрация/вход.
- Репетитор: дашборд, конструктор курса (+ИИ), список учеников, заявки, расписание, биллинг, рассылки, настройки профиля/витрины.
- Ученик: дашборд, мой курс (плеер уроков), задания, прогресс, оплаты, сообщения, созвоны.
- Админ (суперадмин платформы): модерация, пользователи, платежи, аналитика и **Settings**
  (вкладка со всеми бизнес-настройками — см. 4.2).

## 8. Риски и как их снять

| Риск | Снятие |
|------|--------|
| Слишком большой объём (3 продукта сразу) | Жёсткий MVP-срез, D — через интеграции |
| Мы — платёжный посредник (гибрид-модель) | Сплитование ЮKassa (не держим чужие деньги вручную), агентский договор, выплаты только верифицированным самозанятым/ИП. Запуск разбит на 2a/2b |
| Юр-сложности с деньгами и ПДн | Оферта + агентский договор + согласия ПДн **до** включения оплат; ядро (Этап 1) работает без эквайринга |
| «Официальные гос-программы» — неподъёмный контент | В MVP только ИИ-генерация по цели |
| Видео/реалтайм сложны и дороги | Внешние провайдеры, не своя разработка |
| ИИ генерирует ошибки в учебном материале | Человек-в-цикле: ИИ = черновик, репетитор утверждает |
| Нет дизайнера → «как у всех» | Дизайн-система + один яркий фирменный акцент, shadcn как база |
